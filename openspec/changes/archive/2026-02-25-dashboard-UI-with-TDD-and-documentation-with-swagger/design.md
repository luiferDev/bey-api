# Design: Dashboard UI with TDD and Swagger Documentation

## Technical Approach

This design addresses three deliverables from the proposal: (1) Dashboard UI served as static files via Gin, (2) Swagger/OpenAPI documentation using swaggo/swag, and (3) TDD workflow for new handler tests. The dashboard will be a vanilla JavaScript SPA that fetches data from existing `/api/v1/*` endpoints. Swagger docs will be generated from Go annotations in handler files and served at `/swagger/`. TDD follows existing repository test patterns using SQLite in-memory DB.

## Architecture Decisions

### Decision: Dashboard Serving Approach

**Choice**: Serve static files from `cmd/api/static/` directory using Gin `Static()` middleware, with root route redirecting to `index.html`.

**Alternatives considered**: 
- Embed static files using Go's `embed` package — eliminates file system dependency but requires rebuild for UI changes
- Serve from separate static server — adds deployment complexity

**Rationale**: Simple approach matching Gin conventions; no rebuild needed for UI changes during development; files are easily editable by designers.

### Decision: Swagger Generation Tool

**Choice**: Use `swaggo/swag` v1.8.x CLI to generate OpenAPI 3.0 docs from Go annotations in handlers.

**Alternatives considered**:
- Manual OpenAPI YAML/JSON — error-prone, hard to maintain
- go-swagger — generates from code comments but different annotation style

**Rationale**: Industry-standard for Go/Gin projects; annotations colocated with handlers; generates `docs.go` that integrates seamlessly with `gin-swagger` middleware.

### Decision: Testing Framework Approach

**Choice**: Use Go's built-in `testing` package with table-driven tests; SQLite in-memory for repository tests; `httptest` for handler tests.

**Alternatives considered**:
- testify — adds assertion library but introduces new dependency
- ginkgo/gomega — BDD style but steeper learning curve

**Rationale**: Matches existing repository test patterns in `repository_test.go` files; no new dependencies; standard Go practice.

### Decision: Dashboard SPA Routing

**Choice**: Hash-based SPA routing (`/#products`, `/#orders`) to avoid server-side route handling.

**Alternatives considered**:
- HTML5 History API with server-side catch-all — requires middleware to serve index.html for unknown routes
- Query parameter routing (`?view=products`) — less clean URLs

**Rationale**: Works with static file serving without backend changes; browser handles navigation; simple implementation.

## Interfaces / Contracts

### Dashboard API Contract

The dashboard consumes existing API endpoints without modification:

```go
// GET /api/v1/products?offset=0&limit=20
// Response: []Product

// GET /api/v1/orders?offset=0&limit=20
// Response: []Order

// GET /api/v1/inventory
// Response: []Inventory
```

### Swagger Annotations Format

Handlers use standard swaggo annotations:

```go
// GetProducts    godoc
// @Summary      List products
// @Description  Retrieves all products with pagination
// @Tags         products
// @Accept       json
// @Produce      json
// @Param        offset query int false "Offset"
// @Param        limit  query int false "Limit"
// @Success      200 {array} Product
// @Router       /products [get]
func (h *ProductHandler) GetProducts(c *gin.Context) { ... }
```

### Static File Configuration

```yaml
app:
  host: "0.0.0.0"
  port: 8080
  mode: "debug"
  static_path: "./cmd/api/static"  # Dashboard files
  swagger_enabled: true            # Enable/disable swagger
```

## Testing Strategy

| Layer | What to Test | Approach |
|-------|-------------|----------|
| Unit | Handler request/response | Table-driven with `httptest`; mock repositories |
| Unit | Repository CRUD operations | SQLite in-memory; existing pattern from `repository_test.go` |
| Integration | Static file serving | Verify files load via HTTP request |
| Integration | Swagger endpoint | Verify `/swagger/index.html` returns 200 |

## Migration / Rollout

No migration required. This change adds new features without modifying existing database schemas or API contracts.

Phased rollout:
1. Generate Swagger docs first (no UI changes)
2. Verify `/swagger/` accessible
3. Deploy static files
4. Verify dashboard loads
